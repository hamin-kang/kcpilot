# KCpilot 데이터 파이프라인

## 0. 큰 그림 — 우리가 만들려는 게 뭔가?

사용자가 "헤어드라이어 220V 1200W"라고 입력하면, AI가 관련 법령을 찾아서 "안전인증 + 적합등록이 필요합니다"라고 답해야 한다. 그런데 **AI(Gemini)는 한국 KC 법령을 외우고 있지 않다.** 그래서 우리가 법령을 미리 준비해두고, 질문이 들어올 때 **관련 법령 조각을 찾아서 AI에게 같이 건네주는** 방식을 쓴다. 이걸 RAG(Retrieval-Augmented Generation, 검색 증강 생성)라고 한다.

전체 흐름을 그림으로 보면:

```
[준비 단계 — 우리가 미리 하는 일]
원본 데이터          전처리              임베딩            저장
(법령 PDF/API)  →  깨끗한 조각들  →  숫자 벡터로 변환  →  pgvector DB
                  (자르고 청소)      (의미를 숫자로)

[사용 단계 — 사용자가 질문할 때 자동으로 일어나는 일]
사용자 질문  →  질문도 숫자 벡터로  →  DB에서 가장 비슷한 조각 6개 꺼냄  →  AI에게 [질문+조각] 건넴  →  답변
```

**핵심:** 이 문서가 다루는 건 위 그림의 **"준비 단계"** 전체다. 전처리(자르고 청소하고 분류)가 엉성하면, 사용 단계에서 아무리 좋은 AI를 써도 엉뚱한 법령을 꺼내와서 틀린 답을 한다. 데이터 바닥에서 흔히 말하는 **"쓰레기를 넣으면 쓰레기가 나온다(garbage in, garbage out)"** 가 이거다.

이 준비 단계는 raw 전수를 자동으로 파싱·청킹·적재하는 파이프라인으로 구현돼 있다(조문 ~650건 + 품목 ~510건). 실행·코드는 [data_preprocessing_howto.md](data_preprocessing_howto.md)가 SSOT고, 이 문서는 그 바탕 개념을 설명한다.

> **이 문서와 [data_preprocessing_howto.md](data_preprocessing_howto.md)의 역할 분담:** 이 문서는 **개념·데이터 출처·로드맵**을 다룬다. 실제 확정 코드(`preprocess.py`의 `clean_legal_text()`, `ingest.py` 수정, `retrieve_node` 필터)는 그 실습 문서가 SSOT다. 아래 STEP들의 코드는 개념 스케치이며, 함수명·정규식·필터 형태가 다르면 실습 문서를 따른다.

## 1. 임베딩과 검색이 실제로 어떻게 동작하나 (왜 전처리가 결과를 좌우하는지)

전처리를 왜 그렇게 해야 하는지 이해하려면, 먼저 "검색"이 내부에서 어떻게 일어나는지 알아야 한다.

### 1-1. 임베딩 = 문장을 숫자 목록으로 바꾸는 것

**임베딩(embedding)** 은 텍스트를 숫자 목록(벡터)으로 바꾸는 것이다. 우리 프로젝트는 `gemini-embedding-001` 모델로 텍스트 하나를 **3072개의 숫자**로 바꾼다.

```
"헤어드라이어 과열 화재"   →  [0.12, -0.83, 0.41, ... ]  (숫자 3072개)
"모발 건조기 발화 위험"     →  [0.14, -0.79, 0.38, ... ]  (숫자 3072개)
"자동차 타이어 마모"        →  [-0.55, 0.22, 0.91, ... ]  (숫자 3072개)
```

중요한 성질: **의미가 비슷한 문장은 숫자도 비슷하게** 나온다. 위에서 1번과 2번("헤어드라이어 과열"과 "모발 건조기 발화")은 단어는 다르지만 의미가 비슷해서 숫자 목록이 서로 가깝다. 3번(타이어)은 의미가 멀어서 숫자도 멀다.

### 1-2. 코사인 유사도 = 두 숫자 목록이 얼마나 가까운지 재는 값

두 벡터가 얼마나 비슷한지를 **0~1 사이의 숫자 하나**로 잰 게 코사인 유사도다. 1에 가까우면 매우 비슷, 0에 가까우면 무관.

```
"헤어드라이어 과열 화재"  vs  "모발 건조기 발화 위험"   → 0.82  (비슷)
"헤어드라이어 과열 화재"  vs  "자동차 타이어 마모"      → 0.31  (무관)
```

### 1-3. 검색 = 질문 벡터와 가장 가까운 저장 벡터를 꺼내는 것

`ingest.py`로 법령들을 임베딩해서 pgvector에 저장해두면, 각 법령 조각이 3072개 숫자로 DB에 들어가 있다. 사용자가 질문하면:

1. 질문도 똑같이 3072개 숫자로 임베딩한다
2. DB에 저장된 모든 조각과 코사인 유사도를 계산한다
3. **가장 가까운 상위 K개**(우리는 `TOP_K_LAW=6`, 즉 6개)를 꺼낸다

코드로는 `workflow.py`의 이 한 줄이다:

```python
# 질문(query)과 가장 비슷한 법령 6개를 점수와 함께 꺼낸다
pairs = store.similarity_search_with_score(query, k=settings.TOP_K_LAW)
```

### 1-4. ⭐ 그래서 전처리가 왜 중요한가 — 한 조각이 너무 크면 망한다

여기가 핵심이다. **임베딩은 조각 "전체 텍스트"를 하나의 의미 벡터로 뭉뚱그린다**(엄밀히 항목들의 산술평균은 아니지만, 직관적으로는 "여러 주제가 섞여 흐려진다"고 보면 된다).

예를 들어 별표3 파일 하나에 (삭제된 항목을 빼도) 수십 개 품목이 들어 있다(실측 ~30여 개). 이 파일 전체를 임베딩하면, 그 3072개 숫자는 "전선 + 스위치 + 커패시터 + 변압기 + 다리미 + 헤어드라이어 + 냉장고 + ... 수십 개가 한데 섞여 흐려진 의미"가 된다.

이제 사용자가 "헤어드라이어"로 검색하면:
- 별표3 전체 벡터는 수십 개 품목이 뭉뚱그려진 것이라, 헤어드라이어와의 유사도가 **낮게** 나온다 (헤어드라이어는 그중 1개일 뿐이니까) — 예시로 0.6 안팎
- 만약 "헤어드라이어"만 따로 담긴 작은 조각이 있었다면 유사도가 **훨씬 높게**(예시로 0.85+) 나왔을 것이다

품목이 늘면 더 심각해진다. 어린이제품 별표, 생활용품 별표가 추가되면, 검색이 **엉뚱한 별표를 더 높은 점수로** 꺼내올 수도 있다. requirements §4.2.1의 "코사인이 0.6~0.8에 압축된다"는 현상은 본질적으로 임베딩 모델 특성(이방성)이지만, **조각이 너무 크면 정답 조각의 절대 유사도까지 더 끌어내려** 이 문제를 악화시킨다(즉 점수 계산 로직이 아니라 데이터 쪽 원인).

**결론:** 한 조각 = 한 가지 주제가 되도록 잘게 쪼개야 한다. 이게 [STEP 2 청킹](#step-2-청킹--큰-덩어리를-의미-단위로-쪼개기)에서 할 일이다.

## 2. 데이터는 두 종류다 — 어떤 건 아예 검색을 쓰면 안 된다

전처리를 시작하기 전에, 네 데이터가 **성격이 다른 두 종류**라는 걸 구분해야 한다. 종류에 따라 다루는 법이 완전히 다르다.

### A종 — 판정용 데이터 (정답이 정해져 있음)

"모발관리기는 어느 인증 대상인가?" 같은 질문의 답은 **별표에 딱 적혀 있는 사실**이다. 확률이 아니다. 별표3의 `1. 안전인증대상전기용품 → 사. 전기기기 → 5) 모발관리기`를 보면 "안전인증"으로 끝난다.

이런 데이터는 **검색(임베딩)으로 추측하게 하면 안 된다.** 왜냐하면 임베딩 검색은 본질적으로 "비슷한 걸 찾는" 거라서, "모발관리기"를 검색했는데 "모발가습기"나 "전기마사지기"를 비슷하다고 잘못 꺼내올 수 있다. 판정은 틀리면 안 되는데 검색은 틀릴 수 있다.

→ 이런 데이터는 **인증등급을 추론하지 말고 구조화된 값으로 꺼내야 한다.** 핵심은 "RDBMS냐 pgvector냐"가 아니라 cert_level을 본문에서 읽지 않는 것이다. 우리는 품목을 `kc_items` 컬렉션에 넣되, **품목명만 임베딩해 매칭**하고 **cert_level은 metadata에서 그대로 꺼낸다**. 예: "헤어드라이어"로 매칭 → 품목 "모발관리기" → `metadata.cert_level = 안전인증`. 매칭은 의미 검색이 하고(동의어를 건너야 하므로), 등급은 표가 정한다. (RDBMS 품목 마스터 승격은 backend가 품목 목록을 직접 다뤄야 할 때의 후속 작업이다.)

### B종 — 근거/설명용 데이터 (자연어로 설명해야 함)

"왜 그 인증이 필요한가?", "무슨 시험을 거치나?", "비슷한 리콜 사례가 있나?" 이런 건 정해진 표로 답할 수 없고 **법령 문장을 찾아서 보여줘야** 한다. 사용자 질문도 "가정용 모발 건조기"처럼 법령 용어와 다를 수 있어서, 의미로 찾는 임베딩 검색이 적합하다.

→ 이런 데이터는 **pgvector에 임베딩해서 의미로 검색한다.** (법률 조문, 안전기준, 리콜 사례)

### 지금은 어떻게 구현돼 있나

A종과 B종을 **pgvector 컬렉션 2개로 분리**했다.

- **B종(법령 조문)** → `kc_legal`. 본문 전체를 임베딩해 의미로 검색한다.
- **A종(별표 품목)** → `kc_items`. 품목명만 임베딩해 매칭하고, cert_level은 metadata에서 그대로 꺼낸다. 품목 1개 = 조각 1개라 검색 희석도 함께 해결된다.

이러면 A종의 등급 정확성(표가 정함)과 동의어 매칭(의미 검색이 함)을 동시에 얻는다. RDBMS 품목 마스터 승격은 backend가 품목을 직접 다뤄야 할 때의 후속 작업이다.

## STEP 1. 데이터 받기 (다운로드)

원본 데이터를 어디서 어떻게 가져오는지다. 소스마다 난이도가 다르니 쉬운 것부터.

### 원본 보관 위치

받은 원본은 모두 `ai-service/data/raw/`에 종류별로 보관한다. 전처리·청킹이 끝난 산출물은 `data/processed/`에 들어가고, `ingest.py`도 거기만 읽는다(`parse_*.py`가 raw에서 재생성). `raw/`·`processed/` 둘 다 `.gitignore` 처리돼 커밋되지 않는다.

```
data/raw/
├── law_api/      법령 조문+별표 — 국가법령정보 API JSON 응답 (안전관리법·전파법·시행령·시행규칙)
├── recalls/      리콜 — 공공데이터포털 API JSON
└── standards/    안전기준 — KC 60335 등 PDF/HWP 수동 다운로드
```

> **별표는 (안전·어린이 축은) 따로 안 받아도 된다.** `대한민국 현행법령` API 본문 JSON에 별표가 이미 텍스트로 포함돼 온다. 실측(본문까지 확인):
> - **안전**: `safety_rule` 별표 3(안전인증대상)·4(안전확인)·5(공급자적합성)·6(안전기준준수) — 품목 분류 별표 완비. 별표3 본문 8,564자에 모발관리기 포함.
> - **어린이**: `children_rule` 별표 1·2·3 (안전인증/안전확인/공급자적합성 대상 어린이제품) 완비.
> - 따라서 안전·어린이는 별도 `appendix/` 디렉토리 불필요.
>
> ⚠️ **전자파 축은 예외 — 현행법령 API로 안 온다.** 전파법 시행규칙엔 별표가 없고(전부 서식), 시행령 별표에도 품목 분류표가 없으며, 시행령 제77조의2·3은 "국립전파연구원장이 고시한다"는 위임 조문(본문 14~23자)일 뿐이다. **적합성평가 대상 기자재 목록**은 국립전파연구원 **「방송통신기자재등의 적합성평가에 관한 고시」 별표 1**에 있고, 이는 **행정규칙 API(`target=admrul`)** 로 받는다(현행법령 API와 별개). `fetch_laws.py`의 `ADMRULES`에 행정규칙ID로 핀이 박혀 있어 `emc_conformity.json`으로 자동 수집된다.
>
> 행정규칙은 법령과 본문 호출 방식이 다르다: 법령은 법령ID로 바로 현행 본문이 오지만, 행정규칙은 lawService가 **행정규칙일련번호**(개정마다 바뀜)를 받으므로, 행정규칙ID로 검색 → '현행' 일련번호를 골라 → 그 번호로 본문을 받는다.

### 1-1. 법령 조문 — 국가법령정보 OPEN API (제일 중요, 난이도 ★)

API를 쓰면 자동으로 받을 수 있고, 조문 번호·시행일 같은 정보가 구조화되어 같이 온다.

**① API 사용 신청 (1회, 승인까지 1~2일)**

1. `open.law.go.kr` 접속 → 회원가입
2. 로그인 → "OPEN API" → "OPEN API 신청". **공동활용 법령종류**에서 아래 4개만 체크한다. 응답 형식은 **JSON**으로 받는다(파이썬에서 제일 다루기 쉽다). 단, **별표 본문은 HTML만 제공**된다(조문처럼 JSON으로 안 옴).

   지금(D1) 쓰는 것 — 본문과 그 별표는 카테고리가 따로다:
   - **대한민국 현행법령** (목록 JSON + 본문 JSON) — 법률·시행령·시행규칙 **조문**. 핵심.
   - **법령 별표·서식** (목록 JSON + 본문 HTML) — 위 시행규칙의 **별표**(품목 목록). 이 프로젝트 핵심(STEP 2)이라 꼭 같이.

   나중(D4) 쓰지만 미리 신청 — 승인이 1~2일 걸려 또 기다리지 않으려고:
   - **현행 행정규칙** (목록 JSON + 본문 JSON) — 안전기준 고시(KC 60335 등) 본문.
   - **행정규칙 별표·서식** (목록 JSON + 본문 HTML) — 그 고시의 **별표**(시험항목 표 등). 고시는 본문보다 별표에 실제 기준표가 든 경우가 많아 짝으로 신청한다.

   나머지(판례·헌재결정례·법령해석례·조약·**자치법규** 별표·서식·각 부처 법령해석…)는 KC 진단과 무관하니 체크 안 한다.
3. 승인되면 **기관코드(OC)** 가 생기는데, 이건 **로그인 이메일의 @ 앞부분(ID)** 이다. 예: 이메일이 `hamin@gmail.com`이면 OC는 `hamin`.
4. 승인은 신청 후 1~2일. **그래서 지금 미리 신청해두고**, 승인 기다리는 동안 아래 다른 작업을 한다.

**카테고리 ↔ 데이터 대응 — 왜 4개로 갈리나**

| 카테고리 | 들어있는 것 | 본문 형식 | 이 프로젝트에서 |
|---|---|---|---|
| 대한민국 현행법령 | 법률·시행령·시행규칙 **조문** | JSON | "왜 인증이 필요한가" 근거 |
| 법령 별표·서식 | 위 법령(주로 시행규칙)에 붙은 **별표** | HTML | 품목→인증등급 **판정**의 핵심 (STEP 2) |
| 현행 행정규칙 | **고시**·훈령·예규 본문 | JSON | 안전기준(KC 60335) 본문 (D4) |
| 행정규칙 별표·서식 | 고시에 붙은 **별표** | HTML | 시험항목 표 (D4) |

대응은 두 축으로 갈린다 — **① 법령(법률·시행령·시행규칙) vs 행정규칙(고시·훈령)**, **② 본문(조문) vs 별표**. 조문은 텍스트라 JSON으로 깔끔히 오지만, **별표는 원래 파일(HWP/PDF) 기반**이라 본문 API에선 보통 '파일 링크'로만 따라온다 — 그래서 별표 텍스트는 별표·서식 카테고리에서 따로 받는다(본문 HTML).

> **"대한민국 현행법령 하나면 90%"는 절반만 맞다.** 조문(=근거 텍스트)의 **부피**로는 그 API 하나가 대부분이다. 하지만 이 서비스의 **판정 핵심인 별표**는 거기서 텍스트로 안 와서, 부피로는 작아도 제일 중요한 조각이 빠진다. (별표가 텍스트로 따라오는지 링크로만 오는지는 응답을 `print`로 찍어 확정하고 맞춘다.)

**② 호출 방식 — 두 단계로 동작한다**

법령 API는 검색용(`lawSearch.do`)과 본문용(`lawService.do`) 두 개가 있다. 보통 "검색해서 법령 ID를 얻고 → 그 ID로 본문을 받는" 순서다.

```python
# pip 또는 uv로 requests 설치되어 있어야 함 (없으면: uv add requests)
import requests

OC = "hamin"   # ← 너의 기관코드(이메일 ID)로 바꿔라

# --- 1단계: 법령 이름으로 검색해서 법령ID 얻기 ---
search_url = "http://www.law.go.kr/DRF/lawSearch.do"
params = {
    "OC": OC,
    "target": "law",          # 'law' = 법령을 검색
    "type": "JSON",           # 응답 형식 (JSON 권장 — 파이썬에서 다루기 제일 쉽다)
    "query": "전기용품 및 생활용품 안전관리법",   # 찾을 법령 이름
}
res = requests.get(search_url, params=params)
data = res.json()   # 응답(JSON 문자열)을 파이썬 딕셔너리로 바꾼다
print(data)         # ← 처음엔 반드시 출력해서 실제 응답 구조를 눈으로 봐라
```

> ⚠️ **중요:** API 응답의 정확한 키 이름(예: 법령ID가 `"법령일련번호"`인지 `"MST"`인지)과 중첩 구조는 버전에 따라 다를 수 있다. **그래서 위처럼 `print(data)`로 먼저 응답을 통째로 찍어보고**, 거기서 실제 키 이름을 확인한 뒤 아래 파싱 코드를 맞춰라. 이건 데이터 작업의 기본기다 — "문서만 믿지 말고 실제 응답을 확인한다."

```python
# --- 2단계: 얻은 법령ID(MST)로 본문(조문) 받기 ---
service_url = "http://www.law.go.kr/DRF/lawService.do"
params = {
    "OC": OC,
    "target": "law",
    "type": "JSON",
    "MST": "법령일련번호",   # ← 1단계 응답에서 찾은 실제 값으로 교체
}
res = requests.get(service_url, params=params)
data = res.json()

# 딕셔너리를 파고들어 조문을 하나씩 꺼낸다
#   (중첩 경로·키 이름은 응답마다 다르니 print(data)로 확인하고 맞춘다)
articles = data["법령"]["조문"]["조문단위"]   # ← 실제 키 경로는 응답 보고 확인
for jo in articles:
    조문번호 = jo.get("조문번호")
    조문내용 = jo.get("조문내용")
    print(조문번호, 조문내용)
```

**이 API가 좋은 이유:** 조문 단위로 끊어서 주기 때문에, 네가 지금 손으로 만드는 frontmatter(`law_name`, `article`, `effective_date`)를 **응답에서 자동으로 뽑아 채울 수 있다.** 즉 STEP 2(청킹)가 거의 공짜로 된다 — API가 이미 조문 단위로 잘라주니까.

**받을 법령 목록:**

| 축 | 법령 (조문) | 별표 |
|---|---|---|
| 안전 | 전기용품 및 생활용품 안전관리법 + 시행령 + 시행규칙 | 시행규칙 별표(안전인증·안전확인 대상 품목) |
| 어린이 | 어린이제품 안전특별법 + 시행령 + 시행규칙 | 시행규칙 별표(대상 품목) |
| 전자파 | 전파법 + 시행령 + 시행규칙 | (해당 없음) |

- **조문**(가운데 칸)은 위 JSON 코드로 자동 수집된다 (`대한민국 현행법령` API).
- **별표**(오른쪽 칸)는 안전·어린이 축은 시행규칙 본문 JSON에 텍스트로 포함돼 자동 수집되고, 전자파 축은 적합성평가 고시 별표1 PDF를 `parse_emc_pdf.py`로 파싱한다(STEP 2).
- 안전기준 고시(KC 60335 등)는 여기 없다 — `현행 행정규칙`·`행정규칙 별표·서식`에서 받으며, 시험기준 파싱은 D4.

### 1-2. 리콜·인증 사례 — 공공데이터포털 (난이도 ★)

리콜 데이터는 **제품안전정보센터(safetykorea) Open API**에서 받는다. `fetch_recalls.py`가 KC 관련 품목 키워드로 국내 리콜 목록(`/recall/recallList.json`)을 조회하고 `recallUid` 기준으로 중복 제거해 `data/raw/recalls/recalls.json`에 저장한다(현재 1,600여 건). 인증은 HTTP 헤더 `AuthKey`다.

향후 이 수집을 Spring 백엔드에서 하루 1번 자동 동기화하는 것(`@Scheduled`)은 D3 작업이다.

### 1-3. 안전기준 고시 — PDF 수동 다운로드 (난이도 ★★★)

`KC 60335-2-23` 같은 안전기준(시험 항목의 진짜 출처)은 깔끔한 API가 없다. `fetch_standards.py`가 PDF 첨부를 `data/raw/standards/`에 받아둔다. 다만 분량이 방대하고 표가 많아 파싱은 별도 전략이 필요하며, **시험기준 파싱·적재는 로드맵 D4**다(현재는 받아만 둔 상태).

## STEP 2. 청킹 — 큰 덩어리를 의미 단위로 쪼개기

**청킹(chunking)** = 긴 문서를 검색에 적합한 작은 조각으로 자르는 것. [§1-4](#1-4--그래서-전처리가-왜-중요한가--한-조각이-너무-크면-망한다)에서 봤듯이, 한 조각이 한 가지 주제만 담아야 검색이 정확하다. 비유하면 **두꺼운 교과서를 "한 장에 한 개념"씩 적힌 색인 카드로 나누는** 것이다.

조각 크기는 문서 종류마다 다르게 한다:

| 문서 종류 | 자르는 단위 | 왜 |
|---|---|---|
| 법률·시행령·시행규칙 | **조문 1개 = 조각 1개** | API가 이미 조문 단위로 줌. 의미 경계가 자연스러움 |
| **별표(품목 목록)** | **품목 1개 = 조각 1개** ⭐ | 통째 넣으면 희석(§1-4). 이게 1순위 작업 |
| 안전기준(긴 PDF) | **항목/문단 단위** | 너무 길어서 더 잘게. 아래 RecursiveCharacterTextSplitter 사용 |
| 리콜 | 사례 1건 = 조각 1개 | 이미 적절함 |

### 2-1. ⭐ 별표 쪼개기 — 실제 예시로

지금 `02_안전_시행규칙_별표3.md`의 `사. 전기기기` 부분은 이렇게 생겼다(실제 내용):

**Before (`사. 전기기기` 한 항목에만 26개가 나열되고, 별표 전체로는 수십 개 → 희석):**
```
사. 전기기기:
1) 삭제 <2026. 4. 1.>
2) 전기다리미 및 전기프레스기
3) 주방용전열기구
5) 모발관리기
...
26) 그 밖에 1)부터 25)까지의 기기와 유사한 기기
비고) 정격입력이 10kW 이하인 것만 해당하며, 방폭형인 것은 제외한다.
```

이걸 **품목 1개당 파일 1개**로 쪼갠다.

**After (품목별로 분리 → 각 조각이 한 품목만 담음):**

`02_safety_rule_appx3_hair-dryer.md`:
```yaml
---
doc_type: 별표
axis: 안전
category: 전기용품
cert_level: 안전인증
item_code: ELEC-005
item_name: 모발관리기
law_name: 전기용품 및 생활용품 안전관리법 시행규칙
article: 별표 3
effective_date: 2026-04-01
---

안전인증대상전기용품 - 전기기기: 모발관리기.
정격입력이 10kW 이하인 것만 해당하며, 방폭형인 것은 제외한다.
```

이렇게 하면 "헤어드라이어"로 검색했을 때 이 조각의 유사도가 확 올라간다(조각 전체가 모발관리기 얘기뿐이니까). **비고(조건)는 그 품목에 걸리는 것만** 본문에 같이 넣어준다.

**쪼개는 코드 (개념 스케치):** 별표는 형식이 일정해서 정규식으로 품목 줄을 뽑을 수 있다.

```python
import re

# 별표3 '사. 전기기기' 블록에서 품목을 뽑는 예
block = """2) 전기다리미 및 전기프레스기
3) 주방용전열기구
5) 모발관리기"""

# "숫자) 이름" 패턴을 찾되, '삭제'가 들어간 줄은 제외 (정제 — STEP 3와 연결)
for line in block.splitlines():
    m = re.match(r"\s*(\d+)\)\s*(.+)", line)
    if not m:
        continue
    번호, 이름 = m.group(1), m.group(2).strip()
    if "삭제" in 이름:        # 폐지된 품목은 건너뛴다
        continue
    print(f"품목 {번호}: {이름}")
    # → 여기서 이름별로 .md 파일을 생성하면 된다
```

> 핵심 원칙은 "한 조각 = 한 품목"이다. `parse_laws.py`가 분류 별표를 품목 단위로 자동 분리하며, 이 원칙이 검색 정확도를 좌우한다(§1-4).

### 2-2. 긴 문서 쪼개기 — chunk_size와 overlap

안전기준(KC 60335-2-23)처럼 긴 문서는 조문 같은 자연 경계가 없어서, **글자 수 기준**으로 자른다. LangChain이 도구를 준다:

```python
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=800,      # 조각 하나의 최대 글자 수
    chunk_overlap=100,   # 옆 조각과 겹치는 글자 수 (아래 설명)
)
chunks = splitter.split_text(긴_안전기준_텍스트)
# chunks는 ["...첫 800자...", "...다음 800자...", ...] 리스트
```

**chunk_overlap(겹침)이 왜 필요한가?** 글자 수로 무작정 자르면 한 문장이 두 조각으로 끊길 수 있다. 예:

```
겹침 없이 자르면:
  조각1: "...정격 전압이 250V 이하인 것으로 사람 또는"
  조각2: "동물의 피부 또는 모발을 손질하기 위한..."
  → 어느 조각도 "250V 이하 + 모발 손질"을 온전히 못 담음 → 검색 누락 위험

겹침(overlap)을 주면:
  조각1: "...정격 전압이 250V 이하인 것으로 사람 또는 동물의 피부 또는 모발을"
  조각2: "사람 또는 동물의 피부 또는 모발을 손질하기 위한..."  ← 앞부분 겹침
  → 경계 문장이 양쪽에 온전히 남음
```

`RecursiveCharacterTextSplitter`는 가능하면 문단·문장 경계에서 자르려고 시도해서, 무식하게 800자에서 뚝 끊는 것보다 똑똑하다. 시작값은 `chunk_size=800, overlap=100` 정도로 두고, 검색 품질 보면서 조정한다.

## STEP 3. 정제 — 조각을 청소하기

자른 조각들에는 검색을 방해하는 노이즈가 섞여 있다. **임베딩하기 전에** 이걸 제거하고, 쓸모 있는 정보(날짜 등)는 메타데이터로 옮긴다. 네 실제 파일에서 나온 노이즈로 하나씩 보자.

### 3-1. 개정 마커 제거

`01_안전_안전관리법_제5조.md` 본문에 이런 게 섞여 있다(실제):

```
...안전인증을 받아야 한다. <개정 2025. 10. 1.>
```

`<개정 2025. 10. 1.>`는 "이 조항이 언제 바뀌었나"라는 행정 정보라, **법령 의미와 무관하고 검색엔 노이즈**다. 본문에서 빼고, 날짜는 `effective_date` 메타데이터로 옮긴다.

```python
import re

text = "안전인증을 받아야 한다. <개정 2025. 10. 1.>"
clean = re.sub(r"<개정[^>]*>", "", text).strip()
print(clean)   # → "안전인증을 받아야 한다."
```

(`<[^>]*>`가 아니라 `<개정...>`만 지우는 이유: 다른 꺾쇠 내용까지 날리지 않으려고.)

### 3-2. 삭제된 항목 제외

별표3에 이런 줄들이 있다(실제):

```
1) 삭제 <2026. 4. 1.>
4) 삭제 <2026. 4. 1.>
```

이미 **폐지된 품목**이라 인덱싱하면 안 된다(없는 제품으로 잘못 안내할 위험). STEP 2의 쪼개기 코드에서 `if "삭제" in 이름: continue`로 이미 걸렀다. 조문 텍스트 안에 섞인 경우엔 해당 줄을 통째로 버린다:

```python
lines = text.splitlines()
kept = [ln for ln in lines if not re.search(r"삭제\s*<", ln)]
clean = "\n".join(kept)
```

### 3-3. 특수문자·유니코드 정규화

별표에 가운뎃점이 `ㆍ`(한글 자모 영역 문자)로 들어가 있다(실제: `기계ㆍ기구`). 일반 가운뎃점 `·`와 코드가 달라서, 검색·표시가 일관되지 않을 수 있다. 통일한다:

```python
import unicodedata

text = "기계ㆍ기구에 부착되는"
text = text.replace("ㆍ", "·")              # 한글 가운뎃점 → 일반 가운뎃점
text = unicodedata.normalize("NFC", text)   # 유니코드 표기를 표준형(NFC)으로 통일
print(text)   # → "기계·기구에 부착되는"
```

> `NFC`가 뭐냐면: 같은 한글도 컴퓨터 내부 표현이 두 가지("한"을 한 글자로 vs "ㅎ+ㅏ+ㄴ" 조합으로)일 수 있는데, 이걸 한 가지로 통일하는 것이다. 안 하면 "같은 단어인데 검색이 안 되는" 황당한 버그가 생긴다. (이건 네가 전에 물어본 **맥 한글 파일명 NFD 문제와 같은 뿌리**다. 파일명뿐 아니라 본문 텍스트에도 적용한다.)

### 3-4. 비고(조건문) 처리

별표의 `비고)`는 "교류만 해당", "방폭형 제외" 같은 **적용 조건**이다. 버리면 안 되지만 본문 한가운데 섞이면 검색이 흐려진다. 해당 품목 조각의 **맨 뒤에 붙여** 함께 검색되게 한다(STEP 2의 After 예시처럼).

### 정제 요약

| 노이즈 | 처리 | 옮길 곳 |
|---|---|---|
| `<개정 …>` | 본문에서 제거 | `effective_date` |
| `삭제 <…>` 줄 | 통째 버림 | (없음) |
| `ㆍ`, 유니코드 흔들림 | 정규화 | (본문 정리) |
| `비고)` 조건 | 해당 품목 뒤로 | (본문 말미) |

## STEP 4. 분류 태그 달기 (메타데이터)

### 4-1. 메타데이터가 뭔가

**메타데이터(metadata)** = 각 조각에 붙이는 라벨이다. "이 조각은 안전 축이고, 전기용품 카테고리고, 안전인증 단계다" 같은. 우리는 `.md` 파일 맨 위 `---`로 감싼 부분(frontmatter)에 적는다. `ingest.py`가 이걸 읽어서 조각과 함께 DB에 저장한다.

```yaml
---
doc_type: 별표             # 법률 | 시행령 | 시행규칙 | 별표 | 안전기준고시 | FAQ
axis: 안전                 # 안전 | 전자파
category: 전기용품          # 전기용품 | 생활용품 | 어린이제품
cert_level: 안전인증        # 안전인증 | 안전확인 | 공급자적합성확인 | 적합인증 | 적합등록 | 자기적합확인
item_code: ELEC-005
item_name: 모발관리기
law_name: 전기용품 및 생활용품 안전관리법 시행규칙
article: 별표 3
effective_date: 2026-04-01
source_url: https://www.law.go.kr/...
---
```

### 4-2. ⭐ 메타데이터가 검색 정확도를 끌어올리는 법 (필터링)

이게 메타데이터의 진짜 위력이다. 지금 `retrieve_node`는 **모든 법령 조각**에 대해 코사인 검색을 한다. 그런데 사용자 입력 폼에는 이미 `전기 사용=예`, `어린이 대상=아니오`가 들어온다. 그러면 검색하기 **전에** "전기용품/안전·전자파 축만" 남기고 어린이제품 조각을 아예 후보에서 빼버릴 수 있다. 후보가 줄면 엉뚱한 게 섞일 확률이 준다.

비유: 도서관에서 책 찾을 때, 전체 서가를 다 뒤지는 것보다 "과학 코너"로 먼저 좁히고 찾는 게 빠르고 정확한 것과 같다.

**코드 변경 (workflow.py의 retrieve_node):**

```python
# Before — 전체에서 검색
pairs = store.similarity_search_with_score(query, k=settings.TOP_K_LAW)

# After — 카테고리로 먼저 좁히고 검색
#   (langchain-postgres PGVector의 filter 문법. 버전에 따라 형태가 조금 다를 수 있으니
#    네 설치 버전 문서를 확인해서 맞춰라.)
category_filter = {"category": {"$in": ["전기용품"]}}   # 분류 노드 결과에서 동적으로 구성
pairs = store.similarity_search_with_score(
    query,
    k=settings.TOP_K_LAW,
    filter=category_filter,
)
```

이게 동작하려면 **조각을 저장할 때 metadata에 `category`가 들어가 있어야** 한다. 그런데 지금 `ingest.py`는 `category`를 저장하긴 하는데 `doc_type`, `cert_level` 등 새 필드는 안 싣는다 → STEP 5에서 고친다.

> ⚠️ 두 가지 주의:
> - 위 필터는 **최소형**이다. "필터가 너무 좁아 0건이면 필터를 풀고 재검색"하는 안전장치가 붙은 **확정판은 [data_preprocessing_howto.md](data_preprocessing_howto.md) 5단계**에 있다.
> - 별표3 원본은 생활용품 품목(자동차용 재생타이어·가스라이터·압력솥)까지 한 파일에 담아 통째로 `category: 전기용품`으로 태깅돼 있다. 필터가 제대로 동작하려면 STEP 2 청킹에서 **품목별 실제 category**를 넣어야 한다(생활용품 품목을 전기용품으로 두면 필터가 틀린다).

## STEP 5. 임베딩해서 저장하기 (적재)

### 5-1. ingest.py가 실제로 하는 일 (지금 코드 읽기)

`ingest.py`를 실행하면 일어나는 일을 평범한 말로 풀면:

1. `data/processed/*.md` 파일을 전부 읽는다
2. 각 파일을 `---` 기준으로 **frontmatter(메타데이터)** 와 **본문**으로 분리한다 (`_parse_frontmatter`)
3. frontmatter의 **`item_name` 유무로 컬렉션을 가른다**: 없으면 조문(본문 임베딩 → `kc_legal`), 있으면 품목(품목명 임베딩 + cert_level metadata → `kc_items`)
4. 각 조각을 `Document(본문, 메타데이터)` 객체로 만든다
5. **임베딩(3072숫자로 변환)** 해서 해당 컬렉션에 저장한다(Vertex 250건 제한 때문에 200건씩 배치)
6. 리콜 데이터가 있으면 `kc_recalls`에도 저장한다

"컬렉션(collection)"은 그냥 pgvector 안의 **테이블 같은 묶음**이라고 생각하면 된다. 조문은 `kc_legal`, 품목은 `kc_items`, 리콜은 `kc_recalls`로 나눠 담는다.

재실행하면 기존 컬렉션을 지우고 새로 넣는다(idempotent — 몇 번 돌려도 결과가 같음).

### 5-2. 실행

```bash
cd ai-service
uv run python pipeline/ingest.py
```

  [kc_items] 511건 적재 완료
리콜 데이터 없음(data/recalls.json 미존재) — 건너뜀

### 5-3. ingest.py의 정제·메타데이터 처리 (구현 완료, 개념 설명용)

> 아래는 "왜 이렇게 하는가"의 개념 스케치다. **실제 코드는 이미 구현돼 있다** — 정제는 `preprocess.py`의 `clean_legal_text()`, 적재·컬렉션 분기는 `ingest.py`의 `load_documents()`가 SSOT다([data_preprocessing_howto.md](data_preprocessing_howto.md) 참조). 아래 함수명·형태는 옛 스케치라 실제와 다를 수 있다.

핵심 두 가지: ① frontmatter의 메타데이터를 Document에 싣고, ② 본문을 임베딩 전에 정제 함수로 통과시킨다.

**(a) frontmatter에서 새 메타데이터 필드도 싣기.** 지금 `load_law_documents`는 `law/article/axis/category`만 metadata에 넣는다. 새 필드를 추가:

```python
docs.append(Document(
    page_content=body,
    metadata={
        # frontmatter 키는 law_name으로 통일하되, 기존 law도 함께 지원(둘 다 읽음)
        "law": meta.get("law_name") or meta.get("law", "(미상)"),
        "article": meta.get("article", ""),
        "axis": meta.get("axis"),
        "category": meta.get("category"),
        # ↓ 새로 추가하는 필드들
        "doc_type": meta.get("doc_type"),
        "cert_level": meta.get("cert_level"),
        "item_code": meta.get("item_code"),
        "item_name": meta.get("item_name"),
        "effective_date": meta.get("effective_date"),
        "source_url": meta.get("source_url"),
    },
))
```

**(b) 본문 정제 함수를 만들어 임베딩 전에 통과시키기.** STEP 3의 규칙들을 함수 하나로 묶는다:

```python
import re
import unicodedata

def clean_body(text: str) -> str:
    """임베딩 전에 본문에서 노이즈를 제거한다 (STEP 3)."""
    # 1) 개정 마커 제거
    text = re.sub(r"<개정[^>]*>", "", text)
    # 2) '삭제 <...>'가 들어간 줄 제거
    lines = [ln for ln in text.splitlines() if not re.search(r"삭제\s*<", ln)]
    text = "\n".join(lines)
    # 3) 특수문자·유니코드 정규화
    text = text.replace("ㆍ", "·")
    text = unicodedata.normalize("NFC", text)
    # 4) 여러 줄 공백 정리
    text = re.sub(r"\n{3,}", "\n\n", text).strip()
    return text
```

그리고 `load_law_documents`에서 `body`를 쓰기 직전에 `body = clean_body(body)` 한 줄을 넣으면 된다.

> 위 `clean_body`는 개념 스케치다(개정 마커만, 정규식도 단순). **확정판은 `preprocess.py`의 `clean_legal_text()`**([data_preprocessing_howto.md](data_preprocessing_howto.md) 1단계 — 신설/시행일 마커, 삭제줄 앵커 매칭까지 처리)이며, ingest에는 `from preprocess import clean_legal_text`로 연결한다. 함수명이 갈리지 않도록 실습 문서를 따른다.

## STEP 6. 제대로 됐는지 확인하기 (검증)

데이터를 넣었으면 **검색이 잘 되는지 직접 확인**해야 한다. "넣었으니 되겠지"는 금물.

### 6-1. 검색 직접 돌려보기

```python
# ai-service 디렉토리에서: uv run python 으로 실행
from llm import get_vector_store
import settings

store = get_vector_store(settings.LAW_COLLECTION)

# "헤어드라이어"로 검색해서 어떤 법령이, 몇 점으로 나오는지 확인
results = store.similarity_search_with_score("헤어드라이어 가정용 220V", k=5)
for doc, score in results:
    print(f"[{score:.3f}] {doc.metadata.get('item_name') or doc.metadata.get('article')}")
    print(f"        {doc.page_content[:60]}...")
```

**확인 포인트:**
- 헤어드라이어를 검색했는데 **모발관리기 조각이 상위에** 나오는가? (별표 쪼개기가 됐다면 그래야 함)
- 엉뚱한 품목(타이어, 가스라이터)이 위에 끼어 있지 않은가?
- 점수가 0.6대에 뭉쳐 있다면 → 아직 조각이 너무 크다는 신호 → 더 쪼갠다

### 6-2. 정답셋 = 데이터 구멍을 잡는 회귀 테스트

requirements §4.2.1 Phase 3에서 만들 "정답 라벨 10~20건"은 신뢰도 보정용만이 아니라, **데이터가 충분한지 확인하는 테스트**이기도 하다.

각 대표 품목마다 "기대 결과"를 한 줄로 적어둔다:

```
제품명 | 기대 카테고리 | 기대 인증 | 기대 신뢰도 | 정답 근거가 코퍼스에 있나?
헤어드라이어 | 전기용품 | 안전인증+적합등록 | HIGH/MEDIUM | 별표3 모발관리기(안전인증대상) ✓ / KC60335 ✓
```

만약 헤어드라이어를 진단했는데 근거 조항을 못 꺼내오면 → **점수 버그가 아니라 데이터 구멍**이다(그 조각이 없거나, 너무 크거나, 필터가 과하게 걸림). 이렇게 정답셋이 "데이터가 빠졌는지"를 잡아준다.

## 한 장 요약 (개념 잡힌 뒤 빠르게 볼 체크리스트)

데이터를 추가할 때(상세는 [data_preprocessing_howto.md](data_preprocessing_howto.md)의 "신규 데이터 추가 절차"):

1. **받기** — `fetch_laws.py`로 raw에 법령/고시 수집(또는 PDF를 `data/raw/`에 둠).
2. **등록** — 새 법령이면 `parse_laws.py`의 `SLUG_META`에, 분류 별표면 `CLASSIFICATION_APPENDICES`에 항목 추가.
3. **재생성** — `parse_laws.py` → `parse_emc_pdf.py` → `ingest.py` 순서로 재실행(조문/품목 자동 분기 적재).
4. **검증** — `similarity_search_with_score`로 맞는 품목·조문이 상위에 나오는지 확인. cert_level은 매칭 품목의 metadata에서 나오는지 본다.
