# KCpilot 요구사항 정의서

> **프로젝트명:** KCpilot
>

## 1. 프로젝트 개요

### 1.1 프로젝트 정의

중소기업·스타트업이 KC 인증을 받기 위한 사전 진단을 도와주는 AI 기반 웹 서비스. 사용자가 제품 정보를 자연어로 입력하면, AI가 관련 법령·기준을 검색하여 어떤 인증이 필요한지, 어떤 시험 항목을 거쳐야 하는지, 유사 제품 리콜 사례는 무엇인지를 정리해서 보여준다.

### 1.2 해결하려는 문제

중소기업 인증 실태조사(중소기업중앙회) 기준 다음 문제가 입증되어 있다.

- 인증 비용 부담: 44.3%
- 인증 절차 부담: 35.0%
- 인증 취득 기간 부담: 31.6%
- 인증 취득 실패·포기 사유 1위: 까다로운 인증 취득 절차 (29.8%)

기존 정부 시스템 safetykorea.kr은 이미 인증받은 제품을 검색하는 시스템이지, 미인증 제품에 무엇이 필요한지 진단해주는 시스템이 아니다. 사설 컨설팅은 건당 수십~수백만원으로 영세 제조사·스타트업에 부담이다.

### 1.3 타깃 사용자

**1순위 타깃 (Primary):**
- 신제품을 처음 출시하는 스타트업
- 크라우드펀딩(텀블벅, 와디즈)으로 첫 하드웨어 제품 출시하는 1인 창업자
- 해외 제품을 국내에 처음 유통하려는 소규모 수입업자

**2순위 타깃 (Secondary):**
- 기존과 다른 카테고리로 제품을 확장하는 중소제조사 담당자

**비타깃:**
- 동일 품목을 수년째 만들어온 기존 중소제조사 (이미 절차를 안다)
- 인증 담당 직원이 있는 중견·대기업 (자체 처리 가능)

### 1.4 핵심 가치 제안

"제품 정보를 자연어로 입력하면, 어떤 KC 인증이 필요한지 30초 안에 정리된 리포트를 받습니다. 검색에 1~2주 헤매던 일을 30분으로 줄입니다."

### 1.5 차별점

| | safetykorea.kr | 사설 컨설팅 | ChatGPT 검색 | KCpilot |
|---|---|---|---|---|
| 인증 종류 자동 진단 | ❌ | ✅ | △ (부정확) | ✅ |
| 자연어 입력 | ❌ | ✅ | ✅ | ✅ |
| 복수 인증 동시 식별 (안전+전자파+어린이) | ❌ | ✅ | ❌ | ✅ |
| 리콜 사례 사전 경고 | ❌ | △ | ❌ | ✅ |
| 출처 법령 명시 | △ | ✅ | ❌ | ✅ |
| 최신 고시 반영 | ✅ | ✅ | ❌ | ✅ (반자동 갱신) |
| 비용 | 무료 (단 사용 어려움) | 수십~수백만원 | 무료 | 무료~저가 구독 |

가장 결정적인 차별점은 **복수 인증의 동시 식별**이다. 한 제품에 안전인증 + 전자파 인증 + 어린이제품 인증이 동시에 걸리는 케이스를 누락 없이 안내한다. 이게 ChatGPT·검색으로 가장 자주 빠뜨리는 실수다.

## 2. 도메인 정보

### 2.1 KC 인증의 두 축

KC 인증은 크게 두 가지로 나뉜다.

**축 1 — 안전인증** (산업통상부/KATS 관할)
- 안전인증: 화재·감전 등 위해 우려가 매우 큰 제품 (제품시험 + 공장심사 + 정기검사)
- 안전확인: 위해 우려가 있는 제품 (제품시험만)
- 공급자적합성확인: 위해 우려가 낮은 제품 (자체 시험 또는 제3자 시험 후 신고)

**축 2 — 전자파 인증** (과기부/국립전파연구원 관할)
- 적합인증: 위해 우려 큰 기자재
- 적합등록: 위해 우려 중간
- 자기적합확인: 위해 우려 낮음
- 잠정인증: 기준 미정 기자재

**중요:** 한 제품이 두 축에 동시에 걸릴 수 있다. 예: 헤어드라이어 = 안전인증 + 전자파 인증 둘 다 필요.

### 2.2 제품 카테고리

- 전기용품
- 생활용품
- 어린이제품

한 제품이 복수 카테고리에 걸릴 수 있다. 예: 어린이용 전자완구 = 전기용품 + 어린이제품.

### 2.3 시험기관

KC 인증 시험을 수행하는 기관:
- KTL (한국산업기술시험원) — 종합
- KTR (한국화학융합시험연구원) — 화학·환경 강점
- KTC (한국기계전기전자시험연구원) — 기계·전기 강점
- 그 외 다수 민간 시험기관

전자파 인증은 별도로 국립전파연구원(RRA) 지정 시험기관에서 수행.

KATS는 시험기관이 아니라 제도 관리 기관(정부).

## 3. 기능 요구사항

### 3.1 우선순위 정의

- **Must:** MVP에 반드시 포함. 데모 시연에 필요한 핵심 기능.
- **Should:** 시간 허락하면 포함. 발표 완성도 향상.
- **Could:** 향후 확장 영역.

### 3.2 핵심 기능 (Must)

#### [F-01] 회원가입 및 로그인 (Must)
- **Actor:** 사용자
- **설명:** 이메일/비밀번호 기반 회원가입 및 로그인
- **인증:** JWT 토큰
- **비고:** 진단 이력 관리를 위해 필요

#### [F-02] 제품 정보 입력 (Must)
- **Actor:** 로그인한 사용자
- **입력:**
  - 제품명 (필수)
  - 제품 용도 (필수)
  - 전기 사용 여부 (필수)
  - 어린이 대상 여부 (필수)
  - 주요 사양 (자유 텍스트, 선택)
  - 사양서 PDF (선택, 업로드)
- **출력:** 진단 ID 생성, 다음 단계 진행
- **예외:** 필수 필드 누락 시 추가 입력 요청

#### [F-03] AI 기반 인증 진단 (Must)
- **Actor:** 시스템
- **설명:** 입력 정보를 바탕으로 LangGraph 멀티에이전트가 적용 인증 진단
- **처리 단계:**
  1. 카테고리 분류 (전기/생활/어린이)
  2. 적용 인증 식별 (안전인증/안전확인/공급자적합성확인/전자파 인증 등)
  3. 시험 항목 추출
  4. 시험기관 추천
  5. 유사 리콜 사례 검색
  6. 자기비판 노드를 통한 결과 검증
- **출력:** 진단 결과 JSON
- **응답 시간 목표:** 30~60초 이내

#### [F-04] 진단 결과 표시 (Must)
- **Actor:** 사용자
- **표시 항목:**
  - 적용 인증 종류 (복수 가능)
  - 적용 카테고리 (전기/생활/어린이 중 해당)
  - 위해 단계 (안전인증/안전확인/공급자적합성확인)
  - 적용 시험 항목 목록
  - 시험기관 추천 (KTL·KTR·KTC 비교)
  - 유사 리콜 사례 경고
  - **인증별 신뢰도 라벨 (HIGH / MEDIUM / LOW)** — 각 적용 인증 카드에 개별 신뢰도 라벨 표시. 진단 전체에 대한 단일 신뢰도 값은 저장·표시하지 않는다. 백분율 점수도 사용자에게 노출하지 않는다 (산출 로직은 4.2.1 참조).
  - LOW 인증이 1개라도 포함되면 진단 카드/상세 화면 상단에 **"전문가 상담 권장" 경고 배지**를 표시 (인증별 LOW 라벨에서 동적으로 파생되는 표시이며 별도 저장값 아님)
  - 출처 법령 명시
  - **면책 문구 고정 표시**

#### [F-05] 모호 케이스 처리 (Must)
- **Actor:** 시스템
- **설명:** 신뢰도가 낮거나 경계선 케이스인 경우, AI가 단정하지 않고 솔직히 모호함을 표시
- **응답 예시:** "이 제품은 통제번호 X의 일부 기준에 근접합니다. 다만 ○○ 조항과 중첩되어 확신하기 어렵습니다. 한국제품안전관리원 또는 KTL에 직접 문의를 권장합니다."
- **목적:** AI가 거짓 자신감으로 사용자를 오도하지 않도록 함

#### [F-06] 단계별 추론 로그 표시 (Must)
- **Actor:** 사용자
- **설명:** AI가 어떤 단계를 거쳐 결론에 도달했는지 펼쳐서 볼 수 있음
- **목적:** 설명가능성(Explainable AI). 사용자 신뢰도 확보. 발표 시 핵심 시연 포인트.

#### [F-07] 출처 법령 표시 (Must)
- **Actor:** 시스템
- **설명:** 모든 진단 결과에 RAG가 참조한 법령 조항 명시
- **표시 형식:** "근거: 전기용품 및 생활용품 안전관리법 시행규칙 별표 3 제○호"

#### [F-08] 진단 이력 조회 (Must)
- **Actor:** 사용자
- **설명:** 본인이 진행한 진단 이력 목록 조회 및 상세 보기
- **정렬:** 최신순

### 3.3 보조 기능 (Should)

#### [F-09] 사양서 PDF OCR 추출 (Should)
- **설명:** 사용자가 사양서 PDF 업로드 시 OCR로 사양 자동 추출
- **목적:** 입력 부담 감소
- **기술:** pdfplumber + LLM 후처리

#### [F-10] 표시 의무 체크리스트 (Should)
- **설명:** KC 마크, 모델명, 정격, 제조업체, A/S 연락처 등 의무 표시 항목 자동 생성
- **목적:** 표시 의무 누락 방지

#### [F-11] 시험 의뢰서 초안 자동 생성 (Should)
- **설명:** 진단 결과를 바탕으로 KTL 시험 의뢰서 초안을 PDF로 자동 생성
- **목적:** 신청서 작성 부담 감소

### 3.4 확장 기능 (Could)

#### [F-12] 공급자적합성확인서 자동 작성 (Could)
- **설명:** 공급자적합성확인 대상 제품의 경우, 확인서 양식 자동 작성
- **목적:** 자체 신고 서류 작성 부담 감소

#### [F-13] 데이터 갱신 알림 (Could)
- **설명:** 등록한 제품 카테고리에 신규 리콜 사례 또는 고시 개정이 있으면 사용자에게 알림
- **목적:** 사후 모니터링

## 4. 비기능 요구사항

### 4.1 성능

- 진단 결과 응답 시간: 60초 이내 (목표 30초)
- 동시 사용자: 데모 환경 기준 10명 수준
- DB 응답 시간: 단순 조회 1초 이내

### 4.2 신뢰성 (가장 중요)

- **AI 거짓 자신감 방지:** 모든 답변은 RAG로 검색된 법령 텍스트에 근거해야 함. 법령에 없는 내용 생성 금지.
- **출처 표시 강제:** 모든 진단 결과에 출처 법령 조항 명시.
- **신뢰도 표시:** 근거 충실성에 따라 인증별 HIGH/MEDIUM/LOW 신뢰도 표시. 산출 방법은 4.2.1 참조.
- **자기비판 노드:** LangGraph에 자기비판 노드 포함하여 1차 결론 재검토.
- **면책 문구 상시 표시:** 모든 결과 화면 하단에 "본 서비스는 사전 참고용이며 법적 효력이 없습니다" 고정 표시.

#### 4.2.1 신뢰도 산출 로직

**적용 단위:** 진단(assessment)이 아니라 **각 적용 인증(applicable certification)별로** 신뢰도를 산출한다. 한 진단에 여러 인증이 포함될 수 있고, 인증마다 법령 근거의 명확도가 달라 신뢰도도 달라질 수 있기 때문이다.

**진단 전체 신뢰도는 별도로 저장·표시하지 않는다.** 사용자가 보는 것은 인증별 라벨이며, 여러 인증을 단일 값으로 합치는 것은 정보 손실과 거짓 정밀성 위험이 있어 의도적으로 회피한다. UI상 "전문가 상담 권장" 경고는 LOW 인증이 1개 이상 있는지로 동적으로 판단한다.

**산출 방식 — 검색 게이트 + 근거 충실성**

두 신호를 쓰되, **측정 대상이 다르므로 대등하게 합치지 않는다.** 하나는 "관련 법령을 찾았는가"(관련성)를, 다른 하나는 "그 근거가 결론을 뒷받침하는가"(충실성)를 답한다.

1. **검색 관련성** (`ragScore`, 0.0~1.0)
   - 분류 노드가 만든 **검색 질의** ↔ 해당 인증이 **인용한** 법령 조항 임베딩 간 cosine similarity의 평균(인용 근거별 평균). 즉 원입력 전체가 아니라 "검색에 쓴 질의 ↔ 인용된 근거" 사이의 값이다. (구현: `workflow.py`의 `_assemble`에서 `cited_refs`의 similarity 평균.)
   - 답하는 질문: *"관련 법령을 찾기는 했는가?"* — 의미적 **관련성** 신호이며, 결론의 정확성·근거 충실성을 재는 값이 아니다.
   - 임베딩 특성상 관련 문서라도 유사도가 0.6~0.8 대에 압축된다. 이 값을 그대로 신뢰도로 쓰면 정답이라도 HIGH(≥0.85)에 도달할 수 없다(척도 미스매치).

2. **근거 충실성** (`llmScore`, 0.0~1.0)
   - LangGraph 자기비판 노드(F-03, F-05)에서 LLM이 *"인용된 법령이 이 결론을 실제로 뒷받침하는가"* 를 NLI 방식으로 평가한 점수.
   - 답하는 질문: *"이 판단을 믿어도 되는가?"* — 결론의 충실성(faithfulness) 신호. **신뢰도의 주축**이다.

3. **결합 규칙 (검색을 게이트로, 충실성을 주축으로):**
   ```
   인용 근거 없음          → confidenceScore = 0                   # 뒷받침할 법령 없음
   ragScore < FLOOR(0.45)  → confidenceScore = min(llmScore, MEDIUM−0.01) # =0.64, 검색 빈약 → 최대 LOW로 강등
   그 외                    → confidenceScore = llmScore            # 관련 법령 확보 → 충실성이 신뢰도 결정
   ```
   - `ragScore`는 "관련 법령을 못 찾았는가"를 거르는 **게이트**로만 쓴다. 신뢰도 자체는 충실성 판단이 끈다.
   - **기존 `MIN(ragScore, llmScore)` 방식은 폐기했다.** 코사인(관련성, 0.6~0.8 압축)과 LLM 충실성 점수는 척도가 달라 MIN이 늘 코사인을 채택했고, 그 결과 자기비판 노드가 신뢰도에 영향을 주지 못했으며 정답도 HIGH가 불가능했다. (근거: RAGAS 등 RAG 평가 문헌 — 코사인은 relevance 신호이지 faithfulness가 아니다. 환각도 주제만 겹치면 코사인은 높게 나온다.)

**임계값 (보정 전 잠정값):**

| confidenceScore 범위 | 라벨 |
|---|---|
| `>= 0.85` | HIGH |
| `>= 0.65 and < 0.85` | MEDIUM |
| `< 0.65` | LOW |

임계값(0.85/0.65)과 검색 게이트 바닥(0.45)은 아직 **정답셋으로 보정되지 않은 초기값**이다. "숫자는 지어내지 말고 측정한다"는 원칙에 따라 단계적으로 보정한다:

- **Phase 1 — 구조 교정 (완료):** `MIN` 폐기 → 검색 게이트 + 충실성 주축으로 전환.
- **Phase 3 — 임계값 보정 (예정, 우선):** 시나리오 A~E 등 정답 라벨 10~20건으로 임계값·게이트 바닥을 측정·보정. 현재 값은 그 전까지의 잠정값.
- **Phase 2 — reranker 도입 (후속, 코퍼스 확장 시):** 법령 코퍼스가 커지면 LLM/cross-encoder reranker로 calibrated된 관련성 점수를 얻어 두 신호 결합을 재검토. 코퍼스가 작은 현재(법령 6건)는 reranking 이득이 없어 보류.

**노출 정책:**

- DB에는 `confidenceScore` (numeric(3,2))와 라벨을 모두 저장. score는 디버깅·관리자 분석·calibration 재조정용.
- API 응답·UI에는 **라벨만 노출** (`HIGH`/`MEDIUM`/`LOW`). 백분율 점수는 노출하지 않는다.
  - 이유: LLM·임베딩 기반 점수는 calibration이 보장되지 않아 정밀해 보이는 숫자가 사용자에게 거짓 확신을 줄 수 있다. 3단계 라벨은 본질적인 불확실성 수준을 더 정직하게 전달한다.
